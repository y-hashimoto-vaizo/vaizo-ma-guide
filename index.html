SECTION 2

IRRって何？ — エンジニアのためのゼロからの解説

IRR = 「この投資、年利換算すると何%？」

IRR（Internal Rate of Return / 内部収益率）は、投資の「年あたりのリターン」を示す数字。プログラミングでいうと、パフォーマンスベンチマークのスコアのようなもの。異なる投資案件を同じ物差しで比較できる。

コードで考えるIRR

あるAPIサーバーの処理速度を「リクエスト/秒」で比較するように、IRRは投資の効率を「%/年」で比較する指標。サーバーAが500 req/sで、サーバーBが200 req/sなら、Aの方が効率がいい。同じように、案件AのIRRが30%で、案件BのIRRが12%なら、案件Aの方が投資効率がいい。

IRRの計算イメージ

-500万
Year 0: 買収費用
→
+100万
Year 1: 事業の利益
→
+700万
Year 1: 利益+売却

この場合、500万を投じて12ヶ月後に合計800万を回収。IRRは約60%。VAIZOが目指す理想の水準。

SECTION 3

重要：IRRだけで判断してはいけない理由

IRRはあくまで「1つのスコア」。テストカバレッジが100%でもバグだらけのコードがあるように、IRRが高くても危険な案件はある。

IRRの4つの弱点

1. 短期が有利に見える

2年で2倍のIRR = 41%。7年で2倍のIRR = 10%。同じ2倍でも期間で全然違う数字になる。

2. Exit価格に左右される

「最後にいくらで売れるか」を楽観的に見積もるだけでIRRは跳ね上がる。数字のお化粧がしやすい。

3. 金額が見えない

IRR 50%でも投資10万円なら利益5万円。IRR 20%でも投資1000万円なら利益200万円。率だけでは金額がわからない。

4. 計算が不安定になる場合も

キャッシュフローが黒字→赤字→黒字と変動すると、IRRの計算自体が不安定になることがある。

だからIRR以外にも見る指標

FCF（フリーキャッシュフロー）
「実際に手元に残る現金」のこと。会計上の利益（PL）ではなく、現金ベースで判断する。PL上は黒字でも、設備投資や在庫が重ければ現金は残らない。まずFCFを見る。
MOIC（投下資本倍率）
投じたお金が合計で何倍になったか。500万投じて1500万回収ならMOIC = 3.0x。IRRと違い、直感的にわかりやすい「何倍になったか」の数字。
回収期間
投じたお金を何年で取り戻せるか。3年以内なら強い、5年以内は許容、7年以上は慎重に。VAIZOは12ヶ月以内の売却を方針としているため、特に重要。
下振れ耐性（悲観ケース）
最悪の場合に「死なないか」を確認する。テストでいう「異常系テスト」と同じ発想。正常系（楽観ケース）で利益が出るのは当然。異常系でも致命傷にならないことが重要。
SECTION 4

FCF vs EBITDA — 「利益」と「現金」は違う

エンジニア向けのたとえ

GitHubのスター数（EBITDA）が多くても、実際のアクティブユーザー数（FCF）が少ないOSSプロジェクトがある。見た目の評価と実態は別物。事業を買うときも「見た目の利益」ではなく「実際に残る現金」で判断する。

FCFの計算式

FCF = 税引後営業利益 + 減価償却 - 設備投資 - 運転資本の増加額

// 要するに「最終的に手元に残る現金」

PLが黒字でもFCFがマイナスになるケース

項目PL上の見え方現金の実態
売掛金が多い売上は計上済み現金はまだ入ってきていない
在庫が重い資産として計上現金が在庫に化けて寝ている
設備投資が大きい減価償却で分散実際の支払いは一括で出ていく
維持費が継続的利益は出ているメンテコストで現金が流出し続ける

PL（利益）が良くてもCF（現金）が悪い案件は買ってはいけない

SECTION 5

買う案件と買わない案件の判断基準

✗ 買ってはいけない案件

  • EBITDAは高いがFCFが弱い
  • 在庫・設備・売掛が重い
  • オーナーが抜けると崩れる
  • 主要顧客が1〜2社に集中
  • 契約継続に不確実性がある
  • Exit前提でしか成立しない
  • 悲観ケースで即死する
  • 成長・売却の前提が楽観的

✓ 買うべき案件

  • 現金創出力がある
  • 総投下額で見ても安い
  • 下振れしても耐えられる
  • Exitなしでも一定の回収が見える
  • オーナー依存を減らせる
  • 顧客基盤が分散している
  • 自分たちの改善余地がある
  • AIで価値向上が可能
最終判断は
「儲かりそうか」ではなく「外しにくいか」で行う

SECTION 7

買収案件を評価する手順

1

総投下額を出す

買収価格だけではなく、DD費用・仲介手数料・契約費用・PMI初期費用・運転資金追加をすべて合算する。「本当にいくらかかるのか」の全体像を把握。

2

3ケースでFCFを組む

悲観ケース / 標準ケース / 楽観ケース。特に悲観ケースでも致命傷にならないかを最重視。

3

保有期間とExit前提を置く

VAIZOは最長12ヶ月。Exit EBITDA × Exit Multipleで売却想定価格を算出。売れなかった場合でも成立するかを必ず確認。

4

5つの指標を出力する

IRR / MOIC / 回収期間 / 年次FCF / 累積回収額 — この5つをセットで見て総合判断。

5

定性チェック

オーナー依存度・顧客集中度・属人性・AIによる改善余地。数字では見えない「構造的なリスク」をここで確認。

SECTION 8 — 最重要

VAIZOの事業方針：やること・やらないこと

✗ やらないこと

業界特化型Vertical SaaSは開発しない。開発するのは、買収事業に適用して人を増やさず成長させられる、業務プロセス単位のAI自動化のみ。

  • 特定業界に縛られたSaaSを開発する方向には進まない
  • 開発するAIは業務プロセス単位で、業界横断的に適用可能なもの
  • 判断基準：そのAIを買収事業に適用して、人を増やさず急成長させられるか？

✖ 絶対にやらないこと

社員数が何人もいるような製造業などは、絶対に買収しない。

  • 人件費が固定費として重くのしかかり、FCFを圧迫する
  • 労務リスク（解雇規制・社会保険・労働紛争）が極めて高い
  • AIによる自動化の効果が限定的（物理作業が多い）
  • 12ヶ月以内の売却が極めて困難

✓ 優先順位を上げて注力すべき領域

(a) 買収ターゲット

一人社長が経営しており、業務の9割方がAIで代替できるような事業。

  • オーナー依存を排除しやすい
  • 人件費リスクが極小
  • AI導入による劇的な価値向上が見込める
  • 12ヶ月以内の売却に適した規模感
(b) 開発すべきAI

そこで発生している煩雑な業務や事業活動を対象に、AIによる業務の無人化を開発する。

  • 買収先のバリューアップに直結する開発
  • 売却時の企業価値を最大化するためのAI
  • 汎用SaaSではなく、事業特化の無人化
  • 再現性のある開発プロセスを構築
(c) AI開発の最重要KPI

開発するAIが売上を上げ、コストを下げ、残るFCF（フリーキャッシュフロー）を最大化すること。

  • 売上向上：AIが営業・集客・顧客対応を自動化し、売上を伸ばす
  • コスト削減：人件費・外注費・業務コストをAIで圧縮する
  • FCF最大化：上記の結果として手元に残る現金を最大化する
  • FCFの改善 = 売却時の企業価値（EBITDA × Multiple）に直結
